# 设计决策记录

本文档记录 pantyhose TLS 加密隧道功能的所有设计决策，包括每个选型的候选方案、最终选择和理由。

---

## 决策 1：加密隧道的架构模型

**问题**：如何在 SOCKS5 代理基础上加入加密传输？

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **A. 隧道模式（✅ 选定）** | 加密传输层包裹 SOCKS5，SOCKS5 协议运行在隧道内部 | 不侵入现有逻辑，关注点分离，可保留非加密兼容 |
| B. 协议内集成 | 修改 SOCKS5 握手/数据传输流程，在协议内加密 | 破坏标准 SOCKS5 兼容性，修改量大 |

**最终架构**：
```
[应用] → [ProxyBridge] → 127.0.0.1:1080 → [pantyhose-client] ══TLS══> [pantyhose-server] → [目标]
```

**理由**：SNI remap 等现有功能完全不受影响；加密层和代理层解耦，测试更容易。

---

## 决策 2：加密协议选择

**问题**：隧道层使用什么加密协议？

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **A. TLS 1.3 mTLS（✅ 选定）** | Go 标准库 `crypto/tls`，双向证书认证 | 零外部依赖，1-RTT 握手，安全审计历史成熟 |
| B. Noise Protocol | Curve25519 + ChaCha20-Poly1305，类似 WireGuard | 需 `flynn/noise` 依赖，生态工具不如 TLS |
| C. 自定义协议 | 自行设计密钥交换和帧格式 | 灵活但安全风险极高 |

**理由**：Go 标准库原生支持 TLS 1.3 mTLS，与项目"最少依赖"的风格一致。公钥/私钥体系天然契合 mTLS。

---

## 决策 3：证书管理方式

**问题**：mTLS 需要 CA + 服务端 + 客户端证书，如何生成和分发？

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **A. 内置 gencert 命令（✅ 选定）** | `pantyhose-server gencert` 一键生成所有证书 | 降低使用门槛，保持单二进制设计 |
| B. 用户自行生成 | 依赖 openssl / mkcert 等外部工具 | 灵活但门槛高 |
| C. 证书指纹 pinning | 简化模型，无完整 CA 链 | 灵活性低 |

**生成的文件**：
```
certs/
├── ca.crt          ← CA 证书（双方持有）
├── ca.key          ← CA 私钥（仅服务端保管）
├── server.crt      ← 服务端证书
├── server.key      ← 服务端私钥
└── client.pem      ← 客户端捆绑文件（CA cert + client cert + client key，拷贝给客户端）
```

**证书技术细节**：
- 算法：ECDSA P-256
- 默认有效期：3650 天（10 年）
- CA：`IsCA=true`，`MaxPathLen=0`（只能签发终端证书）
- 服务端/客户端证书：`ExtKeyUsage` 同时包含 `ServerAuth` 和 `ClientAuth`

## 决策 4：端口模型

**问题**：启用 TLS 后，服务端端口行为如何？

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **A. 单端口 TLS-only（✅ 选定）** | `--tls` 模式下只接受加密连接 | 最简单，不容易出错 |
| B. 双端口 | 同时提供加密和非加密服务 | 配置复杂 |
| C. 协议嗅探 | 自动判断 TLS vs SOCKS5 | 边界情况多，安全风险 |

**理由**：实际部署中要么全加密要么不加密。如需同时运行，起两个实例即可。

---

## 决策 5：UDP 流量加密

**问题**：TLS 基于 TCP，无法直接加密 UDP 数据报。

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **A. UDP over TCP（✅ 选定）** | UDP 封装进 TLS/TCP 隧道传输 | 简单，零新依赖，企业内网延迟可忽略 |
| B. DTLS | UDP 专用 TLS 变体 | 保留 UDP 语义，但需 `pion/dtls` 依赖 |
| C. 仅加密 TCP | UDP 保持原样 | 安全性不完整 |

**延迟影响分析**（企业内网场景）：
- 正常情况（无丢包）：额外延迟 <0.05ms，不可感知
- 丢包场景（>1%）：TCP 队头阻塞可能导致延迟尖峰。但企业 LAN 丢包率通常 <0.01%
- 主要 UDP 场景为 DNS 查询（短报文单次请求），不受队头阻塞影响

---

## 决策 6：多路复用方案

**问题**：所有流量走同一条 TLS 连接，需要区分多个 SOCKS5 会话。

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **A. hashicorp/yamux（✅ 选定）** | 生产级多路复用库，stream 实现 `net.Conn` | socks5 库零改动，内置流控和心跳 |
| B. 自研帧协议 | `[stream_id + type + length + payload]` | 零依赖但边界情况极多（半关闭、流控、死锁） |

**选择 yamux 而非 smux 的原因**：HashiCorp 在 Consul、Nomad 等生产系统中大量使用 yamux，稳定性有保证。

**选择多路复用而非每连接独立 TLS 的原因**：实际使用中打开一个网页可能产生几十个并发连接，每个都建立 TLS 握手浪费资源。

---

## 决策 7：客户端架构

**问题**：客户端如何实现？是 fork ProxyBridge 还是独立开发？

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **A. 独立 Go 客户端（✅ 选定）** | `pantyhose-client`，仅做隧道转发 | ProxyBridge 不需改动，Go 天然跨平台 |
| B. Fork ProxyBridge | 在 ProxyBridge 中集成 TLS+yamux | ProxyBridge 是 C/C#/Swift 多语言项目，维护成本极高 |

**分工**：
- ProxyBridge：负责进程级流量拦截和路由规则
- pantyhose-client：负责 TLS 隧道加密转发
- pantyhose-server：负责 SOCKS5 代理和 SNI remap

---

## 决策 8：规则引擎

**问题**：pantyhose-client 是否需要实现路由规则？

**结论**：**不需要**。ProxyBridge 已经提供了完善的进程级路由规则管理（PROXY / DIRECT / BLOCK）。pantyhose-client 只做一件事：把本地 SOCKS5 流量加密转发到远程服务端。

---

## 决策 9：代码组织

**问题**：客户端和服务端代码放在同一仓库还是分开？

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **A. 同一仓库 monorepo（✅ 选定）** | 共享 TLS/隧道代码，版本同步 | 单一 go.mod，一次 `go test` 覆盖全部 |
| B. 独立仓库 | 职责分离 | 共享代码需发布为 Go module，联调不便 |

---

## 决策 10：目录结构

**问题**：引入 server/client 双二进制后如何组织代码？

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **A. 完整 cmd/ 结构（✅ 选定）** | `cmd/pantyhose-server/` + `cmd/pantyhose-client/` + `internal/` | 标准 Go 项目结构 |
| B. 最小改动 | 服务端保持根目录 | 结构不对称 |

**理由**：项目从单二进制进化为双组件，一步到位。本身就是 breaking change（删除了认证），不存在"平滑升级"。

---

## 决策 11：SOCKS5 认证

**问题**：mTLS 启用后，SOCKS5 用户名密码认证的去留？

**结论**：**完全移除**，不管是否启用 TLS。

- mTLS 已验证客户端身份，密码多余
- 本地 pantyhose-client 监听 127.0.0.1，只有本机能连
- 减少配置负担（ProxyBridge 不需要配用户名密码）
- 如需多用户区分，为每人签发不同客户端证书

---

## 决策 12：非 TLS 模式

**问题**：移除认证后，非 TLS 模式 = 完全开放代理，安全吗？

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| A. 保留非 TLS + 警告 | 启动时打印安全警告 | 新用户可能忽略 |
| B. 强制 TLS | 不支持非加密模式 | 开发调试不方便 |
| **C. --insecure 显式 opt-in（✅ 选定）** | 默认要求 TLS，`--insecure` 明确确认 | 安全默认，不阻挡高级用户 |

---

## 决策 13：二进制命名

| 方案 | 描述 |
|------|------|
| A. 保持 `pantyhose` | 向后兼容 |
| **B. 对称命名（✅ 选定）** | `pantyhose-server` / `pantyhose-client` |

**理由**：项目已是 breaking change，对称命名在 Release 页面上一目了然。

---

## 决策 14：断线重连

| 方案 | 描述 |
|------|------|
| **A. 自动重连 + 指数退避（✅ 选定）** | 1s → 2s → 4s → 8s → ... → 60s |
| B. 退出进程 | 依赖外部进程管理器 |
| C. 按需建立连接 | 无预连接 |

**理由**：pantyhose-client 是长期后台服务，需无人值守运行。yamux 自带 30s 心跳检测断连。

---

## 决策 15：Docker 测试

**策略**：分层测试，Docker 为可选。

1. `go test ./...` — 快速单元测试 + 集成测试（无需 Docker）
2. `docker compose -f docker-compose.test.yml up` — 容器化完整测试（可选）

---

## 决策 16：gencert 命令归属

**结论**：作为 `pantyhose-server` 的子命令。证书通常在服务端生成，然后拷贝客户端部分到客户端机器，符合使用习惯。
