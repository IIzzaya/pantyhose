# 证书管理指南

## 文件说明

`pantyhose-server gencert` 生成 5 个文件：

```
certs/
├── ca.crt          CA 公钥证书（双方持有，用于验证对方身份）
├── ca.key          CA 私钥（仅服务端保管，签发新证书时使用）
├── server.crt      服务端证书（证明服务端身份）
├── server.key      服务端私钥（永远不离开服务端）
└── client.pem      客户端捆绑文件（CA cert + client cert + client key）
```

**分发规则**：只需要把 `client.pem` 拷贝给客户端。其他 4 个文件留在服务端。

---

## 多客户端共享

`client.pem` 可以分发给多个客户端共用。所有持有同一个 `client.pem` 的客户端在服务端看来身份相同。

对于个人或小团队使用，共享同一个文件是最简单的做法。

**风险**：如果某个客户端的 `client.pem` 泄露，无法单独吊销——只能重新 `gencert`，让所有旧 `client.pem` 全部失效。

---

## 何时需要重新生成证书

| 场景 | 需要重新 gencert？ |
|------|-------------------|
| 服务端 IP 地址变更 | **是**（新 IP 不在证书 SAN 中） |
| 证书过期（默认 10 年） | **是** |
| 怀疑 `client.pem` 泄露 | **是** |
| 新增一台客户端 | 否（拷贝现有 `client.pem` 即可） |
| 服务端重启 | 否 |
| 客户端/服务端版本升级 | 否 |

**关于服务端 IP 变更**：`gencert` 自动检测本机出站 IP 并写入证书 SAN。如果服务端有多个 IP 或域名，用 `--hosts` 补充：

```bash
pantyhose-server gencert --hosts 10.0.0.5,192.168.1.100,proxy.corp.com
```

---

## 重新生成证书的影响

重新执行 `gencert` 会覆盖 `certs/` 目录下的所有文件：

1. **生成全新的 CA**——新旧 CA 之间没有信任关系
2. **所有客户端的旧 `client.pem` 立即失效**——必须分发新文件
3. **服务端需要重启**以加载新证书
4. **没有平滑过渡**——切换瞬间使用旧证书的客户端会断开连接

**操作步骤**：

```bash
# 1. 在服务端重新生成
pantyhose-server gencert

# 2. 重启服务端
#    停止正在运行的 pantyhose-server，重新启动

# 3. 将新的 client.pem 分发给所有客户端
#    替换客户端 certs/client.pem 后重启客户端
```

---

## 安全建议

- `ca.key` 是最敏感的文件。拥有它可以签发任意客户端证书。如果不需要再签发新证书，可以考虑将其备份后从服务端删除
- `client.pem` 包含客户端私钥，传输时使用安全渠道（如 SCP、U 盘），不要通过即时通讯或邮件明文发送
- 证书默认有效期 10 年（`--days 3650`）。如需缩短，使用 `--days` 参数
